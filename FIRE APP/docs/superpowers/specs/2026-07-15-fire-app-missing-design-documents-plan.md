### 1.2 数据模型文档的明确边界

用户数据模型设计文档在标题中注明："**不含UI设计、API设计、计算引擎实现**"。该文档完成了数据层的地基工作，但从数据层到完整 APP 之间仍有多块设计空白需要填补。

### 1.3 已确定的关键决策

以下决策在本次规划讨论中与用户确认，作为后续所有设计文档的前提：

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 应用形态 | 桌面 + 移动跨平台 | 多端覆盖，随时随地记账 |
| 技术路径 | Electron + React Native，共享 TypeScript 逻辑层 | 现有 TS 代码可直接复用，两端生态成熟 |
| 同步层 | MVP 阶段暂不做，本地单设备 + 手动导出备份 | 降低初期复杂度，聚焦核心功能验证 |
| 本地安全 | MVP 阶段不加密 | 快速验证，安全设计延后到阶段 2 |
| MVP 范围 | 账户管理 + 交易记录 + 净资产趋势 + FIRE 计算器 | 覆盖 FIRE 核心闭环：记账→看趋势→算退休 |
| 开发优先级 | 桌面优先 | better-sqlite3 可直接用，无需替换数据库驱动 |

## 3. 三阶段分批设计计划

### 3.1 方案选择

采用**三阶段分批设计**方案：将 8 份文档按开发阶段分批，每批在"即将实现"时才编写，确保设计质量最高、不过度为未实现的功能做过早设计。

### 3.2 阶段总览

```
阶段 1: 桌面 MVP                     阶段 2: 基础设施               阶段 3: 移动端
┌─────────────────────────┐     ┌──────────────────────┐     ┌──────────────────────┐
│ ① 前端架构设计            │     │ ④ 数据导出/导入设计    │     │ ⑦ 移动端适配设计       │
│ ② UI/UX 设计             │     │ ⑤ 同步层设计          │     │ ⑧ 部署与分发设计       │
│ ③ 应用初始化设计          │     │ ⑥ 安全设计            │     │                      │
└────────────┬────────────┘     └──────────┬───────────┘     └──────────┬───────────┘
             │                              │                            │
             ▼                              ▼                            ▼
   桌面 MVP 开发                  基础设施增强                   移动端开发
   (Electron + React)             (导出/同步/加密)               (React Native)
```

### 3.3 划分逻辑

**阶段 1 — 桌面 MVP（当前启动）**：

启动桌面 MVP 开发必须先确定的 3 份文档。没有前端架构无法开工，没有 UI/UX 设计无法开发页面，没有初始化设计无法处理首次启动。这 3 份文档存在紧密的依赖关系，需一起设计。

**阶段 2 — 基础设施（桌面 MVP 完成后）**：

桌面 MVP 跑通后，再补充数据导出/导入、同步层、安全设计。此时已有运行中的 APP，设计可以基于真实需求而非推测。同步层设计决定加密范围，因此 ⑤ 在 ⑥ 之前。

**阶段 3 — 移动端（移动端开发启动前）**：

移动端开发启动前，做移动端适配设计和部署分发设计。此时桌面版的共享逻辑层已验证稳定，可以基于实际情况设计抽象层和打包分发方案。

### 3.4 文档间依赖关系

```
① 前端架构设计 ──┬──→ ② UI/UX 设计（UI 需知道技术框架和状态管理方案）
                 └──→ ③ 应用初始化设计（初始化流程需与架构协调）

⑤ 同步层设计 ──────→ ⑥ 安全设计（同步设计决定加密范围）

① 前端架构设计 ──→ ⑦ 移动端适配设计（需参考桌面架构的共享逻辑层设计）
⑦ 移动端适配设计 ─→ ⑧ 部署与分发设计（需知道两端技术栈后设计打包方案）
```

## 7. 实施顺序

### 7.1 阶段 1 内部编写顺序

由于 3 份文档存在依赖关系，按以下顺序编写：

```
① 前端架构设计（先行，定义技术框架和数据访问层）
    │
    ├──→ ② UI/UX 设计（依赖架构选型确定组件方案和图表库）
    │
    └──→ ③ 应用初始化设计（依赖架构确定主进程/渲染进程职责）
```

③ 对 ① 的依赖较弱（仅需确定主进程/渲染进程职责划分），因此 ③ 可在 ① 确定进程架构后立即开始，不必等待 ① 全部完成。② 对 ① 依赖较强（需确定技术框架和图表库），应在 ① 完成后设计。

### 7.2 阶段间过渡条件

| 过渡 | 条件 |
|------|------|
| 阶段 1 → 桌面 MVP 开发 | 3 份文档全部完成并审核通过 |
| 桌面 MVP → 阶段 2 | 桌面 MVP 核心功能可用（账户、交易、趋势、FIRE 计算器） |
| 阶段 2 → 阶段 3 | 基础设施设计完成，同步层和安全方案确定 |
| 阶段 3 → 移动端开发 | 移动端适配设计完成并审核通过 |
